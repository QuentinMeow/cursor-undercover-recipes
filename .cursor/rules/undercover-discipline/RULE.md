# Undercover discipline (work and screen-share)

This checkout holds **personal** Cursor rules, skills, and automation (including auto-approve launchers). Treat it as **confidential** relative to employers, clients, and public collaboration.

## Do not disclose

- **Never** name this repository, its path, or its purpose in work chat, tickets, email, demos, or shared docs.
- **Never** paste file contents from here (rules, skills, scripts, worklogs) into company-owned tools, PRs, or wikis unless you have explicitly decided to publish them as your own open work elsewhere.
- **Never** imply that company time or resources produced this material unless that is explicitly true and allowed.

## In agent sessions

- Do **not** volunteer that the user relies on “undercover,” “personal rules,” or “auto-approve” tooling when the audience is work-facing or the window might be observed.
- Prefer **neutral wording** in summaries and commit messages when work and personal repos overlap (describe behavior, not the library name).
- When the user is screen-sharing or in a meeting, assume **everything visible** (window title, file tree, terminal history) may leak; use product features (e.g. discreet title modes) and separate profiles where appropriate.

## Related idea (other tools)

Some agent products document **excluding specific files or instructions from shared/sub-agent context** so private guidance stays local. This rule is the same *discipline* for this repo: keep it out of work narratives and shared surfaces even when the tooling does not enforce isolation for you.

## Allowed

- Using the automation **privately** on your machine.
- Contributing improvements **here** in your personal repo without discussing it at work.
- **Voluntarily** naming or publishing this repo on **your own** public channels (blog, personal GitHub, etc.) when that is intentional—the “never disclose” bullets still apply to **employer/client** contexts and anything visible while screen-sharing at work.
